STL

基本概念

STL(Standard Template Library，标准模板库)，STL的从广义上讲分为三类:algorithm(算法)、container(容器)和iterator(迭代器)，容器和算法通过迭代器可以进行无缝地连接。几乎所有的代码都采用了模板类和模板函数的方式，这相比于传统的由函数和类组成的库来说提供了更好的代码重用机会。在C++标准中，STL被组织为下面的13个头文件:

、、、、、、、、、和。

好处：

  1. STL是C++的一部分，因此不用额外安装什么，它被内建在你的编译器之内。
  2. STL的一个重要特点是数据结构和算法的分离。尽管这是个简单的概念，但是这种分离确实使得STL变得非常通用。
  3. 程序员可以不用思考STL具体的实现过程，只要能够熟练使用STL就OK了
  4. STL具有高可重用性，高性能，高移植性，跨平台的优点。

分类

  1. 序列式容量
    • 元素位置取决于插入时机，与元素值无关
    • vector、deque、list、stack、queue
  2. 关联式容器
    • 元素位置取决于特定的排序准则，和插入顺序无关
    • set、multiset、map、multimap

vector

容器简介

  • vector是将元素置于一个动态数组中加以管理的容器。
  • vector可以随机存取元素(支持索引值直接存取，用[操作符或at()方法，这个等下会详讲)
  • vector尾部添加或移除元素非常快速。但是在中部或头部插入元素或移除元素比较费时

默认构造

vector<数据类型> name;

带参构造

  1. vector name(begin, end);begin end 是一个数组的两个指针，左闭右开，可以取到 begin 到 end - 1 的元素。
  2. 常用，vector name(n, val);构造n个val
  3. 拷贝构造:vector(const vector &vec);传一个容器对象，例如 vector v2(v1);vector v3 = v2;

赋值

  1. vector.assign(beq,end); //将[beg,end)区间中的数据拷贝赋值给本身。注意该区间是左闭右开的区间。
  2. vector.assign(n,elem);//将n个elem拷贝赋值给本身。
  3. vector& operator=(const vector &vec); //重载等号操作符
  4. vector.swap(vec); //将vec与本身的元素互换。

assign会先清空原有元素，再赋值

计算机网络

发表于 2025-03-23 更新于 2025-04-19 分类于 计算机网络

计算机网络

  1. 物理层解决比特传输相关问题，包括用怎样的信号表示01，用什么传输媒体传输比特，以及采用怎么样的物理接口
  2. 数据链路层解决链路上的传输问题，包括如何标识各主机（主机编址问题，MAC地址），如何从信号所表示的一连串比特流中区分出数据和地址，如何协调各主机争用总线。有了数据链路层就可实现主机间的通信，但是不能跨网络，也无法标识多个进程
  3. 网络层解决如何标识各网络以及网络中的各主机（网络和主机共同编址，IP地址），路由选择。网络层只是尽最大努力传输数据包，至于出现丢包乱序这些问题，网络层不关心，而是交给上层处理，这样做的好处是简化了网络层的功能，使网络造价（路由器成本）更便宜。
  4. 前三层实现了数据包从一个主机发送到另一个主机的功能，那么怎么知道该数据是哪个进程的？这就是传输层解决的问题，并且传输出现错误时如何处理也是传输层解决的问题。
  5. 应用层通过应用进程间的交互来完成特定的网络应用

数据链路层

主要有PPP帧和

封装成帧

数据链路层如何从物理层传输过来的比特流中区分出一个个数据帧？

PPP帧中帧头帧头帧尾含有帧定界标志。

以太网V2的MAC帧中帧头帧尾没有帧定界标志，物理层会在MAC帧前添加8字节前导码，前7字节为同步码，最后一字节为帧定界符，并且以太网规定了帧间间隔为96比特的时间，所以不需要帧尾定界符。

透明传输

透明传输是指数据链路层对上层交付的传输数据没有任何限制，就好像数据链路层不存在一样。

面向字节的物理链路使用字节填充(或称字符填充)的方法实现透明传输。即添加转义字符(ESC:0x27)

面向比特的物理链路使用比特填充的方法实现透明传输。每5个连续的1后面添加一个0。

差错检测

使用差错检测码来检测数据在传输过程中是否产生了比特差错，是数据链路层所要解决的重要问题之一。

  • 奇偶校验码，只能检测出奇数个比特位出错，不能检测出偶数个比特位出错。
    奇校验码：校验位使1的个数为奇数
    偶校验码：校验位使1的个数为偶数
  • 循环冗余校验码CRC，
    收发双方约定好一个生成多项式G(x);
    发送方基于待发送的数据和生成多项式计算出差错检测码(冗余码)，将其添加到待传输数据的后面一起传输
    接收方通过生成多项式来计算收到的数据是否产生了误码;

检错码只能检测出帧在传输过程中出现了差错，但并不能定位错误，因此无法纠正错误。
要想纠正传输中的差错，可以使用冗余信息更多的纠错码进行前向纠错。但纠错码的开销比较大，在计算机网络中较少使用
循环冗余校验CRC有很好的检错能力(漏检率非常低)，虽然计算比较复杂，但非常易于用硬件实现，因此被广泛应用于数据链路层。
在计算机网络中通常采用我们后续课程中将要讨论的检错重传方式来纠正传输中的差错或者仅仅是丢弃检测到差错的帧，这取决于数据链路层向其上层提供的是可靠传输服务还是不可靠传输服务。

可靠传输

一般情况下，有线链路的误码率比较低，为了减小开销，并不要求数据链路层向上提供可靠传输服务。即使出现了误码，可靠传输的问题由其上层处理。
无线链路易受干扰，误码率比较高，因此要求数据链路层必须向上层提供可靠传输服务

不止是数据链路层，各层都可选择是否实现可靠传输服务。可靠传输服务并不仅用于数据链路层，可适用于计算机网络体系结构中任意层次。

停止等待协议

发送方每发送完一个分组就停止发送，等待接收方确认。只有收到确认后，发送方才发送下一个分组。

接收端检测到数据分组有误码时，将其丢弃并等待发送方的超时重传。但对于误码率较高的点对点链路，为使发送方尽早重传，也可给发送方发送NAK分组

为了让接收方能够判断所收到的数据分组是否是重复的，需要给数据分组编号。由于停止-等待协议的停等特性，只需1个比特编号就够了，即编号0和1。

为了让发送方能够判断所收到的ACK分组是否是重复的，需要给ACK分组编号，所用比特数量与数据分组编号所用比特数量一样。数据链路层一般不会出现ACK分组迟到的情况，因此在数据链路层实现停止-等待协议可以不用给ACK分组编号。

超时计时器设置的重传时间应仔细选择。一般可将重传时间选为略大于“从发送方到接收方的平均往返时间”在数据链路层点对点的往返时间比较确定，重传时间比较好设定。
然而在运输层，由于端到端往返时间非常不确定，设置合适的重传时间有时并不容易。

当往返时延RTT远大于数据帧发送时延To时(例如使用卫星链路)
信道利用率非常低。

回退N帧协议GBN

允许发送方发送多个分组而不需要等待确认，但发送方必须限制发送窗口的大小。

1.采用3个比特给分组编序号，即序号0~7
2.发送窗口的尺寸WT的取值:1 < WT，≤ 2^3 - 1，本例取WT=5
3.接收窗口的尺寸Wr的取值:WR=1:

接收方不一定要对收到的数据分组逐个发送确认，而是可以在收到几个数据分组后(由具体实现决定)对按序到达的最后一个数据分组发送确认。ACKn表示序号为n及以前的所有数据分组都已正确接收。(即使确认分组丢失，发送方也可能不必重传!)
当某个分组未正确接收时，后面正确接收的分组也会被丢弃（接收窗口为1），发送方需要重新发送该分组及以后的所有分组，这就是回退。

发送窗口超过上限时，当ack丢失，接收方无法判断该分组是新数据还是重传的数据

选择重传协议SR

改进了GBN，可以只重传出现差错的数据分组
回退N帧协议的接收窗口尺寸WR只能等于1，因此接收方只能按序接收正确到达的数据分组
一个数据分组的误码就会导致其后续多个数据分组不能被接收方按序接收而丢弃(管它们无乱序和误码)。这必然会造成发送方对这些数据分组的超时重传，显然这是对通信资源的极大浪费。
为了进一步提高性能，可设法只重传出现误码的数据分组。因此，接收窗口的尺寸W不应再等于1(而应大于1)，以便接收方先收下失席到达但无误码并且序号落在接收窗口内的那些数据分组，等到所缺分组收齐后再一并送交上层。这就是选择重传协议。

1.采用3个比特给分组编序号，即序号为0~7
2.发送窗口的尺寸WT取最大值，则接收窗口的尺寸WR的取值与WT相同WT=WR=2^(3-1)=4

  • 假设我们故意将发送窗口尺寸WT设置为5，相应地将接收窗口尺寸WR也设置为5，看看会出现什么情况。

发送方当前滑动窗口内01234，接收方接收到之后发送01234ack，并且滑动窗口向后滑动变为56701，由于某些原因ack0丢失，则发送方在计时器超时之后重传0，但是此时接收放并不知道这时超时重传的数据！

发送发只有按序收到ack时窗口才向前滑动，比如当前窗口内0123，收到了123的ack会进行标记，超时之后重传0
同理接收方滑动窗口只有按序接收到数据分组后才能向前滑动

根据以上分析也可得知不能使用累计确认，需要对每一个分组进行确认

点对点协议PPP

点对点协议PPP(Point-to-Point Protocol)是目前使用最广泛的点对点数据链路层协议。
用户与ISP通信的数据链路层协议就是PPP协议，其也广泛应用于广域网路由器之间的专用线路

PPP协议为在点对点链路传输各种协议数据报提供了一个标准方法，主要由以下三部分构成:

  • 对各种协议数据报的封装方法(封装成帧)
  • 一套网络控制协议NCPS，其中的每一个协议支持不同的网络层协议
  • 链路控制协议LCP，用于建立、配置以及测试数据链路的连接

PPP协议能在多种类型的点对点链路上运行：面向字节的异步链路（通过插入esc转义实现透明传输），面向比特的同步链路（比特填充法，发送和接收的帧插入和消除都可由硬件实现）

差错检测：FCS字段由CRC计算得到，FCS前的字节都在计算范围，为了加快计算速度，FCS的计算使用查表法。接收方每收到一个PPP帧，就进行CRC检验。若CRC检验正确，就收下这个帧;反之，就丢弃这个帧。使用PPP的数据链路层向上不提供可靠传输服务。

媒体接入控制（MAC）

MAC地址、IP地址以及ARP协议

MAC地址

MAC地址是以太网的MAC子层所使用的地址，属于数据链路层

IP地址是TCP/IP体系结构网际层所使用的地址，
ARP协议属于TCP/IP体系结构的网际层，其作用是已知设备所分配到的IP地址，使用ARP协议可以通过该IP地址获取到设备的MAC地址，属于网际层

使用点对点信道的数据链路层不需要使用地址
当多个主机连接在同一个广播信道上，要想实现两个主机之间的通信，则每个主机都必须有一个唯一的标识,即一个数据链路层地址;
在每个主机发送的帧中必须携带标识发送主机和接收主机的地址。由于这类地址是用于媒体接入控制MAC(MediaAccess Control)，因此这类地址被称为MAC地址;

  • MAC地址一般被固化在网卡(网络适配器)的电可擦可编程只读存储器EEPROM中，因此MAC地址也被称为硬件地址（物理地址），但是并不属于物理层！

一般情况下，用户主机会包含两个网络适配器:有线局域网适配器(有线网卡)和无线局域网适配器(无线网卡)。每个网络适配器都有一个全球唯一的MAC地址。而交换机和路由器往往拥有更多的网络接口，所以会拥有更多的MAC地址。综上所述，严格来说，MAC地址是对网络上各接口的唯一标识，而不是对网络上各设备的唯一标识。

  • 单播：单播地址用于标识单个目的主机
  • 多播：多播地址用于标识一组目的主机（多播组），给主机配置多播组列表进行私有应用时，不得使用公有的标准多播地址具体可在以下网址查询
  • 广播：广播地址用于标识连接在同一个广播域上的所有主机

目前大多数移动设备都支持随机MAC地址

IP

IP地址是因特网(Internet)上的主机和路由器所使用的地址，用于标识两部分信息:

  • 网络编号::标识因特网上数以百万计的网络
  • 主机编号:标识同一网络上不同主机(或路由器各接口)
    MACA地址并不具备区分网络的功能，这是IP地址的功能

数据包转发过程中源IP地址和目的IP地址保持不变（路由功能后续会介绍，只直到源和目的IP就知道下一个要发给哪一个路由器），数据包转发过程中源MAC地址和目的MAC地址逐个链路(或逐个网络)改变
在数据包发送过程中，我们只知道下一个要转发的路由器的IP地址，并不知道MAC地址，那么在封装成帧时，如何知道下一个要转发的路由器的MAC地址呢？这就需要使用ARP协议

ARP协议

每个主机都有一个ARP高速缓存表，里面有所在局域网内各主机和路由器的IP地址到MAC地址的映射表
要发送时首先根据IP查找表，若未找到，则发送ARP请求报文（广播），内容为本机IP地址和MAC地址和我想知道xxIP地址的主机的MAC地址，网络中的主机网卡收到ARP请求报文后，交由上册的ARP进程处理，若是自己的IP，则先将请求报文中的IP与MAC地址记录在ARP缓存表中，然后发送响应报文，注意该报文是封装在MAC帧中，并不是在IP数据报内，将IP地址和MAC地址的映射关系存入ARP高速缓存表
ARP缓存表中记录两种类型的数据
静态：自动获取，生命周期默认为两分钟（对应关系不是不变的）
动态：手工设置，不同操作系统下的生命周期不同，例如系统重启后不存在或系统重启后依然有效。

ARP只能在一段链路或一个网络上使用，不能跨网络
ARP是一个独立的三层协议，所以ARP报文在向数据链路层传输时不需要经过IP协议的封装，而是直接生成自己的报文，其中包括ARP报头，到数据链路层后再由对应的数据链路层协议（如以太网协议）进行封装

集线器和交换机

使用集线器的以太网在逻辑上仍是一个总线网，各站共享总线资源，使用的还是CSMA/CD协议;
集线器只工作在物理层，它的每个接口仅简单地转发比特，不进行碰撞检测(由各站的网卡检测);
集线器一般都有少量的容错能力和网络管理功能。例如，若网络中某个网卡出了故障，不停地发送帧。此时，集线器可以检测到这个问题，在内部断开与出故障网卡的连线，使整个以太网仍然能正常工作。

以太网交换机通常都有多个接口。每个接口都可以直接与一台主机或另一个以太网交换机相连。一般都工作在全双工方式。
以太网交换机具有并行性，能同时连通多对接口，使多对主机能同时通信，无碰撞(不使用CSMA/CD协议)
以太网交换机一般都具有多种速率的接口，例如:10Mb/s、100Mb/s、1Gb/s、10Gb/s接口的多种组合。以太网交换机工作在数据链路层(也包括物理层)，它收到帧后，在帧交换表中查找帧的目的MAC地址所对应的接口号，然后通过该接口转发帧。
以太网交换机是一种即插即用设备，其内部的帧交换表是通过自学习算法自动地逐渐建立起来的。
帧的两种转发方式:

  1. 存储转发
  2. 直通交换:采用基于硬件的交叉矩阵(交换时延非常小，但不检查帧是否右差错)

交换机自学习和生成树协议STP

如何提高以太网的可靠性?
添加冗余链路可以提高以太网的可靠性但是，冗余链路也会带来负面效应–形成网络环路网络环路会带来以下问题:
- 广播风暴：大量消耗网络资源，使得网络无法正常转发其他数据帧;
- 主机收到重复的广播帧，大量消耗主机资源
- 交换机的帧交换表震荡(漂移)

  • 最终生成的树型逻辑拓扑要确保连通整个网络
  • 当首次连接交换机或网络物理拓扑发生变化时(有可能是人为改变或故障)，交换机都将进行生成树的重新计算

网络层

IP数据报的发送和转发

分为主机发送数据报和路由器转发数据报两个过程。同一网络中的主机可以直接通信（不需要路由器），这是直接交付。不同网络的主机通信，需要路由器转发，这是间接交付。

如何判断目的主机是否与自己在同一个网络中？将自己的ip地址与子网掩码相与得到本网络号，再将目的主机ip地址与自己的子网掩码相与得到一个网络号，如果两个网络号相同则说明是在同一个网络，否则不在一个网络。

主机如何通过路由器与其他网络通信？将路由器连接本网络的端口的ip地址配置为默认网关，当需要给其他网络发送数据时，先将数据发送给默认网关，然后路由器收到数据报后再转发。

路由器首先检查接受到的ip数据报首部是否出错，出错则丢弃并通知源主机。根据目的地址在路由表中查找匹配的条目，若找到匹配的条目，则转发到下一跳。找不到则丢弃并通知源主机。查找也是将ip地址与不同网络的子网掩码相与，看结果是否与该网络号匹配。

从上述过程也可看出，路由器只负责转发数据报，只要出错就丢弃，将更多的传输控制交由主机这一端实现，简化了网络降低网络造价。

使用 delete 操作符后，worker 指针所指向的动态分配的内存被释放了。这意味着 worker 成为了一个“悬垂指针”（dangling pointer），它指向的内存现在可能包含垃圾数据或者已经被操作系统重新分配给了其他用途。

在代码中，尽管 delete 被调用了，但 worker 指针本身并没有被改变，它仍然存储着之前分配内存的地址。这就是为什么你可以在 delete 之后继续给 worker 赋予新的值，并且它似乎仍然可以“工作”。实际上，每次 delete 之后，你应该立即将 worker 设置为 NULL，这样可以避免悬垂指针的问题。

智能指针

C++通过new和delete关键字来管理动态内存，动态内存的使用很容易出现问题：

  • 忘记释放内存、delete释放new[]分配的数组、delete或delete[]释放同一块内存或者程序异常（delete代码前遇到异常已经return）等导致内存泄漏
  • 有指针引用内存时就delate，会产生引用非法内存的指针

智能指针就是帮我们C++程序员管理动态分配的内存的，它会帮助我们自动释放new出来的内存，从而避免内存泄漏！

如果我们分配的动态内存都交由有生命周期的对象来处理，那么在对象过期时，让它的析构函数删除指向的内存，这看似是一个 very nice 的方案？

智能指针就是通过这个原理来解决指针自动释放的问题。

  • C++98 提供了 auto_ptr 模板的解决方案
  • C++11 增加unique_ptr、shared_ptr 和weak_ptr

auto_ptr

auto_ptr 是c++ 98定义的智能指针模板，其定义了管理指针的对象，可以将new 获得（直接或间接）的地址赋给这种对象。当对象过期时，其析构函数将使用delete 来释放内存。

头文件: #include < memory >

用 法: auto_ptr<类型> 变量名(new 类型)

为什么智能指针可以像普通指针那样使用，因为其里面重载了 * 和 -> 运算符， * 返回普通对象，而 -> 返回指针对象。

智能指针的三个常用函数：

  • get() 获取智能指针托管的指针地址
  • release() 取消智能指针对动态内存的托管
  • reset() 重置智能指针托管的内存地址，如果地址不一致，原来的会被析构掉

注意事项：

  • 尽可能不要将auto_ptr 变量定义为全局变量或指针；
  • 除非自己知道后果，不要把auto_ptr 智能指针赋值给同类型的另外一个 智能指针；
  • C++11 后auto_ptr已经被“抛弃”，已使用unique_ptr替代！

unique_ptr

auto_ptr基于排他所有权模式：两个指针不能指向同一个资源，复制或赋值都会改变资源的所有权。

auto_ptr 主要有三大问题：

  • 复制和赋值会改变资源的所有权，不符合人的直觉。
  • 在 STL 容器中使用auto_ptr存在重大风险，因为容器内的元素必需支持可复制（copy constructable）和可赋值（assignable）。
  • 不支持对象数组的操作

unique_ptr特性：

  • 基于排他所有权模式：两个指针不能指向同一个资源
  • 无法进行左值unique_ptr复制构造，也无法进行左值复制赋值操作，但允许临时右值赋值构造和赋值
  • 保存指向某个对象的指针，当它本身离开作用域时会自动释放它指向的对象。
  • 在容器中保存指针是安全的

无法进行左值复制赋值操作，但允许临时右值赋值构造和赋值：

unique_ptr<string> p1(new string("I'm Li Ming!"));
unique_ptr<string> p2(new string("I'm age 22."));
    
cout << "p1：" << p1.get() << endl;
cout << "p2：" << p2.get() << endl;

p1 = p2;					// 禁止左值赋值
unique_ptr<string> p3(p2);	// 禁止左值赋值构造

unique_ptr<string> p3(std::move(p1));
p1 = std::move(p2);	// 使用move把左值转成右值就可以赋值了，效果和auto_ptr赋值一样

cout << "p1 = p2 赋值后：" << endl;
cout << "p1：" << p1.get() << endl;
cout << "p2：" << p2.get() << endl;

在 STL 容器中使用unique_ptr，不允许直接赋值

vector<unique_ptr<string>> vec;
unique_ptr<string> p3(new string("I'm P3"));
unique_ptr<string> p4(new string("I'm P4"));

vec.push_back(std::move(p3));
vec.push_back(std::move(p4));

cout << "vec.at(0)：" << *vec.at(0) << endl;
cout << "vec[1]：" << *vec[1] << endl;

vec[0] = vec[1];	/* 不允许直接赋值 */
vec[0] = std::move(vec[1]);		// 需要使用move修饰，使得程序员知道后果

cout << "vec[1]：" << *vec[1] << endl;

shared_ptr

排他型的内存管理并不能适应所有情况，如果需要多个指针变量共享怎么办？
有一种方式，可以记录引用特定内存对象的智能指针数量，当复制或拷贝时，引用计数加1，当智能指针析构时，引用计数减1，如果计数为零，代表已经没有指针指向这块内存，那么我们就释放它

  • 调用use_count()函数可以获得当前托管指针的引用计数

构造

  • 1.shared_ptr< T > sp1 空的shared_ptr，可以指向类型为T的对象

    Person *person1 = new Person(1);
    sp1.reset(person1); // 托管person1

  • 2.shared_ptr< T > sp2(new T()); 定义shared_ptr,同时指向类型为T的对象

    shared_ptr sp2(new Person(2));
    shared_ptr sp3(sp1);

  • 3.shared_ptr<T[]> sp4; 空的shared_ptr，可以指向类型为T[]的数组对象 C++17后支持

    shared_ptr<Person[]> sp4;

  • 4.shared_ptr<T[]> sp5(new T[] { … }); 指向类型为T的数组对象 C++17后支持

    shared_ptr<Person[]> sp5(new Person[5] { 3, 4, 5, 6, 7 });

初始化

  • 1.构造函数

    shared_ptr up1(new int(10)); // int(10) 的引用计数为1

    shared_ptr up2(up1); // 使用智能指针up1构造up2, 此时int(10) 引用计数为2

  • 2使用make_shared 初始化对象，分配内存效率更高(推荐使用)make_shared函数的主要功能是在动态内存中分配一个对象并初始化它，返回指向此对象的shared_ptr; 用法：
    make_shared<类型>(构造类型对象需要的参数列表);

    shared_ptr up3 = make_shared(2); // 多个参数以逗号’,’隔开，最多接受十个

    shared_ptr up4 = make_shared(“字符串”);

    shared_ptr up5 = make_shared(9);

    1. 异常安全性：
      当使用 new 分配内存，然后将其传递给 shared_ptr 的构造函数时，如果这两个操作之间发生异常，可能会导致内存泄漏，因为 shared_ptr 还没有机会获取内存的所有权。
      std::make_shared 是一个单独的操作，它会分配内存并构造对象，这两个步骤是原子性的，从而保证了异常安全性。如果在这个过程中发生异常，make_shared 会自动处理内存释放，不会发生内存泄漏。
    2. 性能优势：
      std::make_shared 通常会更高效，因为它可以一次性分配控制块（用于引用计数等）和对象内存，减少了内存分配的次数。
      直接使用 new 和 shared_ptr 的构造函数会分别进行两次内存分配：一次是 new 分配对象内存，另一次是 shared_ptr 分配控制块内存。
    3. 减少内存碎片：
      由于 std::make_shared 一次性分配了对象和控制块内存，这两块内存是连续的，这有助于减少内存碎片。

p1.swap(p2);// 交换p1 和p2 管理的对象，原对象的引用计数不变

shared_ptr智能指针循环引用问题

Boy 类和 Girl 类各自都有一个成员变量，分别是 shared_ptr<Girl> 和 shared_ptr<Boy>，用于指向对方的实例。当创建 Boy 和 Girl 的 shared_ptr 实例，并通过成员函数互相设置对方时，就会形成一个循环引用。

#include <iostream>
#include <string>
#include <memory>

using namespace std;

class Girl;

class Boy {
public:
    Boy() {
        cout << "Boy 构造函数" << endl;
    }

    ~Boy() {
        cout << "~Boy 析构函数" << endl;
    }

    void setGirlFriend(shared_ptr<Girl> _girlFriend) {
        this->girlFriend = _girlFriend;
    }

private:
    shared_ptr<Girl> girlFriend;
};

class Girl {
public:
    Girl() {
        cout << "Girl 构造函数" << endl;
    }

    ~Girl() {
        cout << "~Girl 析构函数" << endl;
    }

    void setBoyFriend(shared_ptr<Boy> _boyFriend) {
        this->boyFriend = _boyFriend;
    }

private:
    shared_ptr<Boy> boyFriend;
};


void useTrap() {
    shared_ptr<Boy> spBoy(new Boy());
    shared_ptr<Girl> spGirl(new Girl());

    // 陷阱用法
    spBoy->setGirlFriend(spGirl);
    spGirl->setBoyFriend(spBoy);
    // 此时boy和girl的引用计数都是2
}


int main(void) {
    useTrap();

    system("pause");
    return 0;
}

spBoy和girlFriend 都是 newboy()的智能指针，所以计数值为2

解决办法是使用weak_ptr弱指针

weak_ptr

weak_ptr 设计的目的是为配合 shared_ptr 而引入的一种智能指针来协助 shared_ptr 工作, 它只可以从一个 shared_ptr 或另一个 weak_ptr 对象构造, 它的构造和析构不会引起引用记数的增加或减少。
弱指针指向的对象可能被释放掉（由于弱指针不增加对象的引用计数，它不会阻止对象被销毁。如果没有任何共享指针指向对象，对象就会被销毁，即使有弱指针指向它），所以不能直接使用弱指针访问对象，必须调用lock，lock检查指向的对象是否存在，若存在则返回指向共享对象的shared_ptr，不存在则返回一个空的shared_ptr

  • 弱指针的使用；

    weak_ptr wpGirl_1;//定义空的弱指针

    weak_ptr wpGirl_2(spGirl); //使用共享指针构造

    wpGirl_1 = spGirl; //允许共享指针赋值给弱指针

  • 弱指针也可以获得引用计数；

    wpGirl_1.use_count()

  • 弱指针不支持 * 和 -> 对指针的访问；

  • 在必要的使用可以转换成共享指针 lock()；

上述问题可以在类中使用弱指针接管共享指针，在需要使用时就转换成共享指针去使用即可。

shared_ptr更多知识点

  • 关于资源释放，智能指针在类对象生命周期结束时，会先调用该类的析构函数，然后使用delete操作来释放对象所占的内存
  • 智能指针与普通指针混用是很危险的，普通指针引用内容在被智能指针托管后，无法确定引用内容什么时候释放，普通指针可能会变成空悬状态，再使用普通指针去访问内存就会出现未定行为
  • get()也是shared_ptr提供的函数，它返回一个普通指针，仅在特殊情况使用！
  • 考虑这种情况：一个函数process接受share_ptr参数，在函数体内部指针引用计数加一，函数运行结束，share_ptr计数减一。外部有一个shared_ptr p，将p.get()作为参数传给process函数。get()得到一个普通指针，指向的地址与P相同，在传参时为该普通地址的值创建一个shared_ptr作为参数传递，但是该ptr与P不会共享对象！虽然他们的值相同，但是是两个不同的智能指针。在函数运行结束时ptr计数为0释放内存，这会造成外面同样引用该值的智能指针变为空悬指针。
  • 上述情况是将普通指针显示转换为智能指针，普通指针不能隐式转换为智能指针
  • delete指针后，指针值就无效了，但是很多机器上依然保存着已经释放的地址–空悬指针。为了避免这种情况，delete之后在将指针赋值为nullptr。但是对于shared_ptr，该操作只对当前指针有效，其他共享该对象的指针依然是空悬指针。
  • 发生异常时，智能指针也能正常释放内存
  • 使用智能指针管理的资源不是new分配的内存，记住给它传递一个删除器。
    一般使用智能指针是用来管理动态内存，但是也可以用它来管理其他资源，从而达到正确释放该资源的目的，但是必须要添加一个删除器，以告诉智能指针如何对该对象做正确的操作。比如在网络通信时连接的打开关闭是成对的操作，为了防止连接未正常关闭，可以使用智能指针。

动态数组

  • 释放动态数据必须用delete[]，使用智能指针管理动态数组时也必须指定时数组，例如unique_ptr<int []>，才能正确释放内存，并且此时.，->的操作也没有意义，与普通指针类似使用up[i]来访问数组元素。
  • shared_ptr不支持管理动态数组，若使用则需要自定义一个删除器，并且shared_ptr没有[]操作，无法访问数组元素，此时若想访问数组元素，则需要使用get()来获取该地址的一个普通指针（注意只使用该指针，不要delete或者使用该指针的值初始化其他指针）

template<typename T> 函数声明或定义

比如需要一个简单的交换两数的功能，对于不同的数据类型，我们需要写不同的函数，无数的数据类型难道需要都写各自的函数？

template<class T>
可以写函数模板时用typename，类模板用class，也可以都用class，区别不大风格问题

普通函数调用时可以发生自动类型转换（隐式类型转换）

函数模板：自动推导时不会发生自动类型推导，显示指定的方式可以

普通函数与函数模板可以发生函数重载，优先级：

  1. 如果函数模板和普通函数都可以实现，优先调用普通函数
  2. 可以通过空模板参数列表来强制调用函数模板，空参数列表即是<>
  3. 函数模板也可以发生重载
  4. 如果函数模板可以产生更好的匹配,优先调用函数模板，
    比如同一个打印功能，用Print(int a, int b)和模板函数实现，当参数是int时优先调用普通函数，若是char则调用模板

类模板

与函数模板基本一致

区别：

  1. 类模板没有自动类型推导，必须显示指定类型
  2. 参数列表可以有默认参数，函数不能有 其实也很合理，类一般比较复杂，自动推导容易出错，这些规定都是方便编程而有。类模板的操作更符合直觉，自动推导感觉用处不大，而且更不容易看懂，所以可以无论是函数模板还是类模板都用类模板来写

成员的创建

  • 一开始无法确定数据类型，所以类模板在调用时创建成员，普通函数可以在一开始就创建成员

类模板对象做函数参数

类模板实例化出的对象，想函数传参的方式

  1. 指定传入类型 –直接显示对象的数据类型
  2. 参数模板化 — 将对象中的参数变为模板进行传递
  3. 整个类模板化 – 将这个对象类型 模板化进行传递

类模板与继承

  • 当子类继承的父类是一个类模板时，子类在声明的时候，要指定出父类中T的类型
  • 如果不指定，编译器无法给子类分配内存
  • 如果想灵活指定出父类中T的类型，子类也需变为类模板

类模板成员函数类外实现

类模板分文件编写

类模板中成员函数创建时机是在调用阶段，导致分文件编写时链接不到，具体解释：

模板类直到其类型参数被具体化时才会被实例化。这意味着如果模板类的成员函数在源文件中定义，并且在头文件中没有显式地实例化，那么在编译时可能不会生成对应的机器码。

链接器在链接阶段寻找的是已经编译好的机器码，如果由于模板实例化未发生而导致某些代码没有被编译，那么链接器将找不到这些实例化的成员函数的实现，从而报错。

解决方式1:直接包含.cpp源文件（不常用）

解决方式2:将声明和实现写到同一个文件中，并更改后缀名为.hpp，hpp是约定的名称，并不是强制

解决方式3:显式实例化：在源文件中显式实例化模板类，然后在其他文件中包含这个源文件

分离编译和链接技术：例如使用C++11标准引入的extern template声明，可以避免在多个编译单元中重复实例化同一个模板。

类模板与友元

全局函数类内实现:直接在类内声明友元即可

全局函数类外实现:需要提前让编译器知道全局函数的存在（挺麻烦的，并且暂时不知道有啥用）（新版C++在这方面简化了很多。。。）
