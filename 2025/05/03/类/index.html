类

基本思想是数据抽象和封装，要实现这些，首先需要定义一个抽象数据类型，由类的设计者思考内部实现，使用者只关心类的作用以及相应接口；

数据抽象：将实现与接口分离开

常量成员函数

可读取类中的数据成员，但不能修改。const可以看作是修饰this指针，从而将指向非常量的this声明为指向常量的指针，所以函数内对成员是只读的。

const可以引发重载

类作用域和成员函数

类本身就是一个作用域，类的成员函数定义嵌套在类的作用域之内。

编译器分两步处理类：先编译成员的声明，然后才是成员函数体；所以在函数体内可以随意使用类中其他成员而无需关注他们的次序；

定义类相关的非成员函数

类需要的辅助函数，如read，print等，虽然也是类的接口，但实际上不属于类本身。一般也是将声明与定义分开，在类内声明，以告诉用户有哪些接口可用。定义可以在类外。

构造函数

C++11新标准规定可以为数据成员提供一个类内初始值；如果类内有内置类型复合类型的成员，那么只有所有成员全都赋予了类内初始值，才适合用默认构造函数。

当我们没有声明构造函数时，编译器自动构建一个默认构造函数；也可以使用 = default来表明默认行为（C++11新特性）；与其他函数一样，如果 = default在类的内部，则是内联的，否则不是；

内联inline

内联是指将函数代码直接插入调用函数的地方展开，而不是进行函数调用，普通函数调用是在调用点插入跳转到函数代码的指令。

定义在类中的函数默认是内联

特点：

  • 建议性：inline关键字对编译器来说是一个建议，编译器可以选择忽略这个建议。是否真正内联取决于编译器的判断和优化策略。
  • 开销：对于小型的、频繁调用的函数，内联可以减少函数调用的开销，提高程序运行效率。
  • 局限性：内联函数会增加编译后的代码大小，因为函数体被复制到每个调用点。因此，对于大型函数或调用不频繁的函数，内联可能并不划算。

引申-列表初始化

一、概念

C++11新特性，列表初始化（List Initialization）是一种通过使用大括号 {} 来初始化变量的方法。

二、特点

  1. 统一初始化：无论是数组、对象、内置类型还是容器，列表初始化都可以用统一的语法来初始化。
  2. 防止窄化（Narrowing）转换：与传统的初始化方式（如圆括号初始化）不同，列表初始化会阻止那些可能导致数据丢失的“窄化”转换。

三、写法

四、应用场景

  1. 防止隐式转换：传统的圆括号初始化允许发生可能丢失数据的转换，而列表初始化会阻止这种窄化转换。窄化转换是指将一个较大的类型（如 double）转换为较小的类型（如 int），这可能会丢失数据。为了避免这种潜在的错误，列表初始化在遇到窄化转换时会生成编译错误。
  2. 数组和容器初始化：提供简洁的语法来初始化容器和数组。
  3. 结构体和类的初始化：提供统一的语法来初始化对象的成员变量（构造函数初始值列表）。

注意：

  • 如果没有给出初始化值，使用列表初始化的对象会被默认初始化。
  • 当使用列表初始化时，类成员或对象的初始化顺序与声明顺序一致。这与传统构造函数初始化列表的顺序一致。

拷贝、赋值和析构

C++中，对于一个含有基本数据类型的类，对两个对象使用赋值运算符（=）时，编译器会自动进行相应数据的拷贝。这种拷贝是成员-wise的，即逐个成员地进行拷贝。

因为vector等容器

  1. 重载了 = 实现了深拷贝
  2. 这些容器负责管理其内部资源的生命周期，包括分配和释放内存。在拷贝过程中，它们会正确地管理这些资源，确保不会出现内存泄漏或重复释放等问题。

所以含有vector等容器的类，拷贝。赋值能够正常工作；当对象被销毁时，将销毁vector对象；

访问控制与封装

类的定义使用 class 和 struct 都可以，它们唯一的区别就在于默认访问权限：

  • struct 定义在第一个访问说明符之前的成员是 public 的
  • class 定义在第一个访问说明符之前的成员是 private 的

另外：

  • class默认的继承访问权限是private；struct默认的继承访问权限是public。
  • struct可以用于定义联合体，而class不能。

友元

类可以允许其他类或者函数访问它的非公有成员，通过友元（friend）实现；

友元只能在类内声明，但具体位置不限。友元函数能定义在类的内部，这样的函数是隐式内联的；

成员函数作友元

要注意声明与调用顺序，必须按照以下结构设计程序（Window_mgr::clear()作为Screen的友元函数）：

  1. 首先定义 Window_mgr::clear() 类，其中声明 clear() 类，但是不能定义它！在 clear() 使用 Screen 的成员前必须声明 Screen；
  2. 接下来定义 Screen，其中包含对 clear() 的友元声明；
  3. 最后定义 clear()，此时它才可以使用 Screen 的成员；

友元声明与作用域

类和非成员函数的声明不是必须在它们的友元声明之前，当一个名字第一次出现在友元声明中，隐式的假定该名字在当前作用域是可见的。当然友元本身并不真的声明在当前作用域中。

友元声明的作用是影响访问权限，它并非真正意义上的声明；

即使在类的内部定义了该友元函数，也必须在类的外部提供相应声明从而使得该函数可见；
换句话说，仅仅用声明友元的类的成员调用该友元函数，即使在内部定义过，也必须提供相应声明；

类的其他特性

可变数据成员

mutabl关键字用于修饰类的成员变量，允许在const成员函数中修改该成员变量的值。

应用场景：

  1. 缓存：例如，一个计算密集型的成员函数可以缓存其结果，以便下次调用时直接返回，而不需要重新计算。
  2. 状态标志：如用于表示对象是否已经被初始化的标志。
  3. 计数器：如记录函数调用次数的计数器。

注意事项：

  • mutable关键字只适用于类的成员变量，不能用于全局变量或局部变量。
  • 使用mutable时要小心，因为它打破了const成员函数不修改对象状态的承诺，可能会使代码的逻辑变得复杂或难以理解。

返回 *this 的成员函数

特点​​：

  • 返回类型是 MyClass&（当前类的引用）
  • 返回 *this（当前对象本身）
  • 允许链式调用：obj.doSomething().doAnotherThing()

注意 const 成员函数返回 *this，返回的是常量引用：、

  • 可以调用​​其他 const 成员函数​​（不修改对象）。
  • 不能调用非 const 成员函数​​（可能修改对象）。

const 重载

同时提供 const 和非 const 版本的成员函数）是一种常见的设计模式，主要用于：
​​- 支持 const 对象的只读访问​​。
​​- 允许非 const 对象的读写访问​​。

当在某个对象上调用成员函数时，根据该对象是否使用 const 决定应该调用哪个版本

类的类型

即使两个类中的成员完全相同，他们也是不同的类；

类的声明可以和定义分开，即前向声明；在前向声明之后定义之前是一个不完全类型，此时我们知道该class A;是一个类型，但是并不知道它有哪些成员。

对于一个类，在创建它的对象前该类应该被定义过而不是仅仅声明。否则编译器不知道该对象需要多少存储空间；类似的，在定义之后才能引用或者指针访问其成员。

所以类中不能包含有自身的对象，因为无限递归，永远无法计算该类的大小。但是类中可以包含自身的引用或指针，或者包含自身的静态类型；

类的作用域

一个类就是一个作用域，所以类外定义成员函数就需要提供类名和函数名。因为在类外成员的名字被隐藏了。

一旦遇到类名，定义的剩余部分就在类的作用域内了，包括参数列表和函数体。但是返回类型是在类名之前的，不在类的作用域内，所以返回类型必须指明是哪个类的；

名字查找与类的作用域

名字查找的过程：

  • 首先，在名字所在的块中寻找其声明语句，只考虑在名字的使用之前出现的声明。
  • 如果没找到，继续查找外层作用域。
  • 如果最终没找到匹配的声明，则程序报错；

定义在类内部的成员函数来说，解析其中名字与上述查找规则有所区别：

  • 首先，编译成员的声明。
  • 知道类全部可见后才编译函数体。

类成员声明的名字查找

上述两阶段的查找方式只适用于成员函数中使用的名字。

声明中使用的名字，包括返回值或参数列表中使用的名字，都必须在使用前确保可见。当编译器在类成员声明中遇到一个名字（如返回类型、参数类型）时：

  • 先查找该类作用域内该名字之前的声明​​（即类中​​当前语句之前​​的部分）。
  • 如果找不到，再向外层作用域（如命名空间、全局作用域）查找​​。
  • 不会考虑类中该名字之后的声明​​（即类内声明顺序影响可见性）。 当编译器看到 balance 函数的声明时，它将在 Account 类的范围内寻找匹配的声明，编译器只考虑类中使用 Money 之前出现的声明。没有找到接着向外层作用域查找，会查找到 typedef。

另一方面，函数体在整个类可见后才会处理，所以 balance 返回的是 Money bal 而非外层作用域的 string。

类型名要特殊处理

一般来说，内层作用域可以重新定义外层作用域中的名字，即使该名字已经在外层作用域使用过。但是在类中不可以，如果成员使用了外层作用域的某个名字，而改名字代表一种类型，则类中不可以对其进行重新定义。

编译器并不负责该行为，即使有风险可能也会通过编译。为了避免这种情况，应该将类中类型名的定义放在类的开始处。

成员定义中普通块作用域的名字查找（成员函数内的名字查找）

  • 首先在成员函数内查找该名字的声明。和之前一样，改名字被使用之前的声明才被考虑；
  • 没找到则继续在类内查找，这时类的所有成员都可以被考虑
  • 类中也没有，则在成员函数定义之前的作用域继续查找。在这一步中，若成员函数定义在类外，则不仅要考虑类定义之前的全局作用域，还要考虑成员函数定义之前的全局作用域（可以简单理解为.h文件和.cpp文件）。
